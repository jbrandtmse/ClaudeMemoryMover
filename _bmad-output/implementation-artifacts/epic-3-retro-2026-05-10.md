## What Went Well

### 1. Fully automated pipeline — zero project lead escalations
Epic 3 ran entirely hands-off. The `epic-cycle` orchestration (dev spawn → code review → commit) completed all 5 stories without requiring project lead intervention. This is the target state for the BMAD workflow.

### 2. 100% Epic 2 retro follow-through via Story 3.0
Structuring the deferred-cleanup story before feature work began meant Epic 3's feature stories (3.1–3.4) started from a clean, correctly documented, and locally testable baseline. The pattern should be repeated for every epic.

### 3. Scan → Remap → Apply phase separation
Clean architectural decomposition. Story 3.1 (read-only scan) was independently testable and shippable. Story 3.2 (collect decisions) consumed 3.1's exports cleanly. Story 3.3 (apply) had full backup-first + collision-check safety. Each story added a testable slice without breaking the previous.

### 4. `readSettingsFileStrict` pattern
Introducing a strict reader that returns `undefined` (ENOENT) | `object` (success) | `'malformed'` (parse failure) was a genuine design breakthrough. The pattern was first established in Story 3.0, then consumed in Story 3.3 to abort the apply phase before any rename on malformed `.claude.json`. It replaces the legacy silent-collapse behavior that treated parse failure as "file not found."

### 5. Integration tests fast and comprehensive
Story 3.4 budgeted 3–8 seconds test runtime; achieved 320ms. The `seedClaudeTree` fixture pattern (always produces a MISSING project, no extra setup) paid off immediately and made test intent obvious. 8 integration scenarios covering all AC cases including collision, cross-OS, dry-run, and memory-only fallback.

### 6. Code review caught real safety issues
10 MEDIUM findings across the epic, all caught in review and patched before commit. Notable catches:
- Intra-batch slug collision (two decisions → same target slug, neither on disk yet — silent data loss without this check)
- Malformed `.claude.json` silently overwritten before rename loop (corrupt config → no recovery path)
- `parseRemapSpecs` validation running in interactive mode where `--remap` is ignored (interactive runs aborted on valid scripted-only flags)
- `buildSuggestion` accepting regular files as remap candidates (suggested non-directories)

## Challenges and Lessons Learned

### 1. Headline lesson: fail loudly and early, before any side effects
Every MEDIUM finding in Epic 3 was a variation on the same theme — a code path that failed silently when it should have thrown a loud, early error:
- Malformed config treated as missing → silently overwritten
- Intra-batch collision not detected before rename loop started
- Dry-run progress line claiming "Renamed:" when no rename occurred
- `buildSuggestion` proposing regular files as remap targets

**Carry forward:** Every story AC that involves filesystem mutation should include an explicit criterion for "aborts loudly before any side effects if preconditions fail."

### 2. Pre-flight checks need to cover more than you think
Story 3.3 had on-disk collision detection, which was correct. Code review identified that intra-batch collision detection was also needed — two decisions mapping to the same target slug before either touches disk. This class of bug (batch-level invariant vs. per-item invariant) is easy to miss. Pre-flight should check both on-disk state AND intra-batch consistency before any mutation begins.

### 3. Dry-run transparency must be end-to-end consistent
The dry-run contract (no writes, no backup, record-what-would-happen) was correctly implemented in the WriteGate but a progress message in Story 3.3 still said "Renamed:" during dry-run. The invariant needs to be checked end-to-end: every user-visible output line, not just the underlying write operations.

### 4. ESLint architectural rules are easy to forget mid-implementation
Story 3.2's dev used `os.homedir()` directly, tripping the `no-restricted-imports` rule. The workaround (`dirname(claudeDir)`) was reasonable but introduced a pre-existing MEDIUM (wrong home for non-default `CLAUDE_CONFIG_DIR`). Dev agents should run `npm run lint` between implementation steps, not only at the end.

### 5. Test mocks must match production option signatures
Story 3.1's `readdir` mock ignored the `withFileTypes` flag, meaning future calls without the flag would silently receive Dirent objects instead of strings. Test mocks that don't mirror production option handling cause silent test rot. Typed mocks (`typeof fn`) catch this class of drift.

---

## Key Insight

**The `readSettingsFileStrict` pattern is the architectural norm to adopt.** Every reader that can encounter malformed files should distinguish three outcomes: absent (ENOENT → `undefined`), present-and-valid (→ parsed object), present-and-malformed (→ `'malformed'` sentinel). The legacy silent-collapse (`undefined` for both ENOENT and parse failure) is a bug waiting to happen in any code path that performs mutations based on the read result. `applyMcpConfig`, `applyTeams`, and `applyPlugins` still use the legacy reader — Epic 4 is the right time to migrate them.
