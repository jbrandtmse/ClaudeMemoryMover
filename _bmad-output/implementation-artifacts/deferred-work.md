## Deferred from: code review of story-1.2 (2026-05-09)

All findings below are real enforcement gaps but were intentionally deferred because the Story 1.2 spec mandated *verbatim* "exact content" for each rule file; patching scope expansion in-place would violate the spec contract. Each item should be reconsidered as part of a future hardening story (candidate: a "rule hardening" story under Epic 1 cleanup, or rolled into the first story that touches each rule's call sites).

- **MEDIUM — `no-process-env-home` misses destructuring and computed-env access.** Bypass patterns: `const { HOME } = process.env;` walks past the `MemberExpression` visitor; `process['env'].HOME` is rejected because the rule requires `node.object.property.type === 'Identifier'` (computed access yields a `Literal`). Resolution path: add `VariableDeclarator` + `ObjectPattern` walker for destructuring and relax the env-access type check to accept the `'env'` `Literal` form. Add tests for both bypass patterns.
- **MEDIUM — `no-console-outside-output` does not block `console.info/debug/trace/table/dir`.** AC3 lists exactly `log/error/warn` plus the two `process.std*.write` forms, so the rule correctly matches AC text — but the architectural intent is that ALL stdout/stderr writes route through the Output module. Resolution path: expand `BANNED_IDENTIFIERS` to cover the full ban list, update AC text in a follow-up story, add tests.
- **LOW — `no-raw-json-parse` does not catch `JSON['parse'](text)` (computed property access).** Theoretical bypass; would only matter if a contributor deliberately works around the rule. Resolution path: relax the `callee.property.type === 'Identifier'` constraint to also accept a `'parse'` string `Literal`. Add a regression test.
- **LOW — `no-hardcoded-separator` does not catch template-literal separators (`` `${a}/${b}` ``).** Spec literally says "string literal whose value is exactly `'/'`", so template strings are out of scope by design. Resolution path: extend rule to walk `TemplateElement` cooked values when the next holes look path-shaped, or accept the gap as an acknowledged limitation in the rule's docs. Best paired with adoption of the path engine (Story 1.7+) so the gap is closed by call-site discipline rather than the lint rule.

## Deferred from: code review of story-1.6 (2026-05-09)

- **LOW — `gunzipSync` has no `maxOutputLength` set; gzip-bomb risk on untrusted bundles.** A tiny gzip stream can decompress to GB and OOM the process. Story 1.6 ACs do not require resource-bounded decompression; cmemmov reads local files (low attacker-control surface). Resolution path: in a hardening pass (or when bundle-import lands in Stories 2.x for shared bundles from the network), call `gunzipSync(bytes, { maxOutputLength: <cap> })` and convert the resulting `RangeError` into a `CmemmovError({ code: 'BUNDLE_INVALID_SCHEMA', hint: 'bundle exceeds max decompressed size' })`. Add a fixture/test using a known gzip-bomb pattern.
- **LOW — Multi-issue Zod errors collapse to first issue.** `parseBundle` reports only `cause.issues[0]?.message` in the hint. If a malformed bundle has several validation problems (e.g., wrong type AND extra field AND missing field), only one is surfaced — fix-and-retry workflow is slower for users. Resolution path: when the Output module's structured-error rendering is wired up (Story 1.10+), have it consume `error.cause.issues` directly and render all issues. The parser's hint can stay terse; the UI layer handles aggregation.
- **LOW — Hex case-sensitivity on the `integrity` checksum comparison.** `createHash('sha256').digest('hex')` returns lower-case; the comparison at line 50 is case-sensitive. A future tool emitting upper-case hex (e.g., a third-party signer) would fail integrity check on otherwise-valid bundles. Resolution path: lower-case the comparison (`bundle.integrity.toLowerCase() === computed`). Defer until a real interop case appears — premature today.
- **LOW — Static `older-bundle-version.cmemmov` checksum is fragile to BundleSchema field-order changes.** The fixture's stored `integrity` was computed against the V8 insertion order produced by Zod's current schema definition. Reordering keys in `BundleSchema` would silently invalidate the fixture's checksum (test would fail with a misleading "corrupt" message). Resolution path: regenerate fixtures via a `beforeAll` helper (already discussed in story Dev Notes) that uses `serializeBundle` itself, OR add a `tests/fixtures/bundles/_generate.ts` script kept in-tree (Story 1.6 deleted it after one-shot use). Best paired with the first story that touches the schema.
- **LOW — Memory: large-bundle serialization holds three full copies in RAM simultaneously** (compact canonical, indented JSON, gzip output). For a 100MB bundle this is ~300MB peak. Story 1.6 does not target streaming. Resolution path: when streaming bundle I/O is on the roadmap (NOT currently in any epic), switch to a streaming canonicalizer + streaming gzip via `zlib.createGzip()`. No action this sprint.
- **LOW — `BUNDLE_VERSION_MISMATCH` error code defined but unused.** `src/core/error.ts:5` declares `BUNDLE_VERSION_MISMATCH` with `exitCode: 2`, but Story 1.6 AC4 explicitly downgrades version mismatch to a warning — so no path raises this code. Resolution path: either remove the code in a Story 1.4 cleanup, or repurpose for a stricter `--strict-version` opt-in flag in a future import command. Tracking only — no immediate action.

## Deferred from: code review of story-1.7 (2026-05-09)

- **MEDIUM — `deepMerge` Set-dedup applies uniformly to every array, but Story Dev Notes only specify string-array dedup for permission-rule arrays.** Object arrays (e.g., hook arrays in `settings.json`) get reference-based dedup which produces duplicates rather than logical-equality dedup, since `new Set([objA, objB])` retains both even when their fields are identical. Resolution path: introduce a per-key dedup strategy table (string arrays → Set, object arrays → identity-by-key, or wholesale replace) when Story 2.x merges land richer settings shapes. Or accept duplicates and let users clean up — they're cosmetic, not corrupting.
- **MEDIUM — Corrupt `settings.json` / `plugins.json` is silently treated as `{}` by `applyGlobalSettings` / `applyMcpConfig` / `applyPlugins`.** `safeParseJson` returns `undefined` on parse failure, the writer treats that as an empty object, deep-merges with the incoming data, and writes — destroying the prior content. Mitigated in practice by the import-time backup invariant (rollback recovers the corrupt-but-original file), but the writer should refuse to write when prior state can't be reasoned about. Resolution path: `applySettingsAt` / `applyMcpConfig` / `applyPlugins` should distinguish "absent" (ENOENT) from "malformed" (parse failure) and throw `CmemmovError({ code: 'INTERNAL', hint: 'settings.json is malformed; restore from backup or fix manually' })` for the latter. Add a regression test using a corrupt-JSON fixture.
- **MEDIUM — `applyPlugins` always writes `plugins.json` even when the existing installation uses the `plugins/<name>/config.json` directory form.** The reader (`readPlugins`) prefers `plugins.json` over the dir, so once `plugins.json` is written, subsequent reads silently mask the directory contents. Story Task 9 line 157 mentioned "same pattern as customCommands/teams depending on format found on disk" but Epic 1's writer always picks the JSON file form. Resolution path: a follow-up plugins-format clarification story decides whether the writer detects existing format and routes accordingly, or whether `plugins.json` is the canonical form going forward and the dir form is migrated.
- **MEDIUM — `applyTeams` merge has no defined behavior for incoming team configs lacking an `id` field.** Currently always written (no merge key → no collision detection). Resolution path: spec clarification — either reject id-less configs at the writer, or fall back to dir-name-based dedup, or always write (current behavior). Add a test case once decided.
- **LOW — `resolveOriginalPath` issues unbounded `Promise.all(stat)` over every JSONL in a project's sessions dir.** No concurrency cap. Acceptable for typical Claude installs (tens of sessions) but could hit EMFILE on very large session histories. Resolution path: introduce a small concurrency limiter (e.g., `p-limit` style) when sessions counts grow problematic, or short-circuit by streaming the dir entries and stopping at the first jsonl whose mtime is newer than a sliding cap. Defer until a real perf complaint surfaces.

Resolution path: PM/Architect ratification — either confirm the exemption as intended (most likely; matches established project test pattern) and amend AC6 to explicitly allow `*.test.ts`, or split test fixtures into `tests/integration/` so the rule can apply strictly. - **LOW — backup-service.ts pruning may include non-backup entries that happen to live in `<claudeDir>/backups/cmemmov/`.** `readdir` returns ALL entries; `rm(..., { recursive: true })` removes them regardless of type. A stray `.DS_Store`, `Thumbs.db`, or accidental file in the backup root would be deleted. Resolution path: filter `entries` to only include directory names matching the timestamp pattern (`/^\d{4}-\d{2}-\d{2}T.*-\d+-[0-9a-f]{8}$/`) before pruning. Low risk because the directory is owned by cmemmov and shouldn't contain other entries; revisit if a user reports unexpected file deletion.
- **LOW — backup-service.ts cross-volume cp fallback can leak the staging directory.** If `rename(staging, backupDir)` rejects with EXDEV and the subsequent `cp(staging, backupDir, { recursive: true })` then throws (e.g., disk full mid-copy), `rm(staging, ...)` is not called and the staging dir orphans in `os.tmpdir()` (`backup-service.ts:50-52`). The OS cleans tmpdir periodically so this is bounded but not zero. Resolution path: wrap the cp call in try/finally so the staging dir is removed regardless of cp's outcome. Defer to a hardening pass — not a Story 1.5 blocker.
- **LOW — `recordedOps()` returns the live internal array reference.** The TypeScript `readonly WriteOp[]` annotation is compile-time only; callers can `as WriteOp[]` to mutate, breaking the gate's invariant that ops are immutable post-recording. Resolution path: return `Object.freeze([...ops])` or a `structuredClone(ops)` per call. Cost: per-call array copy. Defer until a real consumer demands runtime immutability — internal-only API for now.
- **LOW — `backup-service.test.ts:89-109` "does NOT prune when the new backup write throws" exercises an early-throw path (non-existent claudeDir → mkdtemp/cp throws), not a mid-function throw after the new backup is partially written.** The test demonstrates the property in spirit ("any throw before pruning short-circuits it"), but the trickiest variant — backup-write throws AFTER the rename but before pruning — is uncovered. Resolution path: add a fault-injection test using `vi.mock('node:fs/promises')` to force a mid-function throw on `readdir`, asserting that the just-created backup remains and prior entries are intact. Defer to a hardening pass.
